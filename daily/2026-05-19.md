  - `vector_store.py` - 向量数据库（需要 ChromaDB）

## 💡 Key Insights / 关键洞察

1. **RAG 的核心价值**：通过检索外部知识，LLM 可以回答训练数据之外的问题，减少幻觉，提供可追溯的信息来源

2. **技术选型的灵活性**：遇到依赖问题时，可以用简化方案（关键词匹配）替代复杂方案（向量数据库），先理解核心概念

3. **国内 API 选择**：DeepSeek API 在国内可直接访问，API 格式兼容 OpenAI，是 Claude 的良好替代

4. **实践优先**：不要被工具和依赖问题阻塞，先用最简单的方式实现核心功能，理解原理后再优化

5. **🔥 TF-IDF 的局限性发现**（重要！）：
   - **问题**：发现 "Embedding 向量嵌入有什么用途？" 和 "Embedding 向量嵌入" 相似度只有 0.35-0.75
   - **表面**：看起来几乎一模一样
   - **实际**：TF-IDF 认为相似度较低
   - **原因**：
     - TF-IDF 是词袋模型，只看字符/词的匹配，不理解语义
     - 无法识别近义词（"用途" vs "作用"）
     - 无法识别翻译关系（"Embedding" vs "向量嵌入"）
     - 文档长度敏感，长短文档相似度偏低
   - **验证**：创建了 `tfidf_limitation_simple.py` 演示 5 个案例
   - **最关键案例**："Embedding 是什么" vs "向量嵌入是什么" 相似度仅 0.15（几乎不相关！）
   - **解决方案**：需要深度学习 Embedding（如 sentence-transformers）来理解语义

6. **向量检索的本质**：
   - 文档向量化 → 查询向量化 → 相似度计算 → 排序 → 上下文增强 → 生成
   - TF-IDF 适合：快速原型、英文单语、关键词精确匹配
   - 深度学习 Embedding 适合：语义理解、多语言、生产环境

7. **批判性思维的重要性**：
   - 不盲目接受结果，质疑"为什么相似度这么低"
   - 通过实验验证假设
   - 理解算法的工作原理和局限性
   - 这是优秀工程师的特质
