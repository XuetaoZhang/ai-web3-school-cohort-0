## 💡 核心收获

### 1. 先理解原理，再用 Framework

**关键认知**：不要直接用 Framework，先手写一遍。

**原因**：
- Framework 隐藏了底层细节，难以调试
- 不理解原理就用 Framework，遇到问题无法解决
- 手写一遍后，才能理解 Framework 的价值

**实践验证**：
- Day 2-3：手写了 RAG 系统（向量检索、相似度计算）
- Day 4：手写了 Context 管理（分层设计、优先级控制）
- 现在理解了 LlamaIndex 的 Index/Query Engine 在做什么
- 现在理解了 LangChain 的 Memory 在解决什么问题

### 2. 按需引入，保持简单

**关键认知**：不要一开始就用全套 Framework。

**原因**：
- 简单任务不需要复杂方案
- 过度设计会增加维护成本
- Framework 版本迭代快，API 变化频繁

**实践策略**：
- 先用原生 API（OpenAI SDK、Anthropic SDK）
- 遇到重复代码再考虑 Framework
- 只引入需要的组件，不要全量导入

### 3. Frameworks 的价值与成本

**价值**：
- 快速开发：不用从零开始
- 最佳实践：避免常见陷阱
- 生态系统：丰富的插件和工具

**成本**：
- 抽象成本：隐藏了底层细节，难以调试
- 过度设计：简单任务可能不需要 Framework
- 版本迭代快：API 变化频繁

**权衡**：
- 原型阶段：不用 Framework，快速验证
- 生产阶段：按需引入 Framework，提高可维护性

### 4. Web3 场景的特殊挑战

**挑战 1：链上数据集成**
- 问题：LangChain 默认不支持 Web3 数据源
- 解决：需要自己实现 Web3 Tool（RPC 调用、合约查询）

**挑战 2：交易签名**
- 问题：Agent 如何安全地签名交易？
- 解决：需要设计 Human-in-the-loop 流程（Day 4 的 Context 分层）

**挑战 3：实时性**
- 问题：链上数据变化快，缓存策略如何设计？
- 解决：实时查询（余额、授权状态）vs 缓存（合约信息、安全规则）

**挑战 4：Gas 优化**
- 问题：如何减少不必要的链上查询？
- 解决：批量查询、Multicall、缓存策略

## 💭 深度思考

### 思考题 1：如果用 LangChain 重写 RAG 系统，会有什么不同？

**当前实现**（Day 2-3 手写）：
- 手动实现向量存储（Chroma）
- 手动实现检索逻辑（相似度计算）
- 手动实现 Prompt 拼接

**用 LangChain 重写**：
- 使用 `VectorStore`（抽象层，支持多种向量数据库）
- 使用 `Retriever`（封装检索逻辑）
- 使用 `RetrievalQA Chain`（自动拼接 Prompt）

**优势**：
- 代码更简洁（50 行 vs 200 行）
- 更容易切换向量数据库（Chroma → Pinecone）
- 内置最佳实践（Prompt 模板、错误处理）

**劣势**：
- 难以调试（不知道 Chain 内部做了什么）
- 难以定制（比如 Day 4 的 Context 分层）
- 性能开销（抽象层的代价）

### 思考题 2：如果用 LangChain 重写 Agent，会有什么不同？

**当前实现**（Day 4 手写）：
- 手动实现 Context 分层（6 层结构）
- 手动实现工具调用（RPC 查询、模拟交易）
- 手动实现决策逻辑（风险评估）

**用 LangChain 重写**：
- 使用 `Agent`（自动决策和工具调用）
- 使用 `Memory`（自动管理对话历史）
- 使用 `Tools`（封装 RPC 调用）

**优势**：
- 自动决策（不用手写 if-else）
- 自动工具调用（不用手动解析 LLM 输出）
- 内置 Memory 管理

**劣势**：
- 无法实现 Day 4 的 Context 分层（LangChain 的 Memory 是线性的）
- 无法控制决策逻辑（Agent 是黑盒）
- 无法实现 Human-in-the-loop（需要自己扩展）

**结论**：Web3 场景的 Agent 需要高度定制，LangChain 不适合。

### 思考题 3：什么场景下你会选择不用 Framework？

**场景 1：原型验证**
- 需要快速验证想法
- 不确定需求是否成立
- 不用 Framework，直接用 API

**场景 2：高度定制**
- 需要完全控制底层逻辑（比如 Day 4 的 Context 分层）
- Framework 无法满足需求
- 手写更灵活

**场景 3：性能优化**
- 需要极致性能（比如高频交易 Agent）
- Framework 的抽象层有性能开销
- 手写更高效

**场景 4：简单任务**
- 单次 API 调用
- 不需要 Memory、Tools、Chains
- 用 Framework 是过度设计
